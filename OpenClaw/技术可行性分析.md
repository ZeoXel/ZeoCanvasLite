# AI短剧生产系统 - 技术可行性分析

> 基于场景模拟的功能实现可行性调研
> 分析时间：2026-03-05

---

## 一、核心功能模块分解

### 模块 1：飞书生态集成

#### 1.1 多维表格操作

**需求：**
- 读取剧本表、分镜表
- 写入素材链接、状态
- 实时监听表格变化

**技术方案：**
- 飞书开放平台 API：`/open-api/bitable/v1/`
- 支持读取、写入、订阅变更

**可行性：✅ 高**
- 飞书 API 文档完善
- OpenClaw 已有飞书集成基础（extensions/feishu）
- 需要扩展：表格操作模块

**工作量：** 3-5 天

**风险：**
- API 频率限制（需要实现请求队列）
- 大批量写入性能（需要批量接口）

---

#### 1.2 云文档上传

**需求：**
- 上传图片/视频到飞书云文档
- 获取可分享链接
- 在表格中插入链接

**技术方案：**
- 飞书云文档 API：`/open-api/drive/v1/`
- 支持文件上传、权限设置

**可行性：✅ 高**
- API 支持完整
- 文件大小限制：单文件最大 20GB（视频足够）

**工作量：** 2-3 天

**风险：**
- 上传速度（大文件需要分片上传）
- 存储配额（需要监控）

---

#### 1.3 机器人消息交互

**需求：**
- 接收飞书群消息
- 发送进度通知
- 富文本卡片消息

**技术方案：**
- 飞书机器人 API
- Webhook 接收消息
- 卡片消息模板

**可行性：✅ 高**
- OpenClaw 已实现飞书机器人基础
- 需要扩展：卡片消息、进度更新

**工作量：** 2-3 天

**风险：**
- 消息频率限制
- Webhook 稳定性


---

### 模块 2：ZeoCanvas 集成

#### 2.1 ZeoCanvas API 能力调研

**需求：**
- 提交生成任务
- 查询任务状态
- 获取生成结果
- 取消/重做任务

**技术方案：**
- 方案 A：ZeoCanvas 提供 REST API（理想）
- 方案 B：修改 ZeoCanvas 源码添加 API 层
- 方案 C：Puppeteer 自动化控制（备选）

**可行性：⚠️ 待确认**
- 需要调研 ZeoCanvas studio 分支的 API 能力
- 如果无 API，需要评估二次开发工作量

**工作量：**
- 方案 A：3-5 天（集成现有 API）
- 方案 B：10-15 天（开发 API 层）
- 方案 C：7-10 天（Puppeteer 方案）

**风险：**
- ZeoCanvas 架构不支持 API（需要大改）
- Puppeteer 方案稳定性差

**关键决策点：** 必须先调研 ZeoCanvas 的 API 能力

---

#### 2.2 画布实时状态推送

**需求：**
- 生成进度实时更新到画布
- 错误信息实时显示
- 多用户同步查看

**技术方案：**
- WebSocket 实时推送
- Server-Sent Events (SSE)
- 轮询（备选）

**可行性：✅ 高**
- 标准 Web 技术
- ZeoCanvas 如果是 Web 应用，易于集成

**工作量：** 5-7 天

**风险：**
- 并发连接数限制
- 网络断线重连

---

#### 2.3 画布参数调整界面

**需求：**
- 滑块调整参数（运动强度、速度等）
- 实时预览
- 参数变更通知 OpenClaw

**技术方案：**
- 前端：React/Vue 组件
- 后端：参数变更 API
- 通知：WebSocket 或 Webhook

**可行性：✅ 高**
- 标准前端开发

**工作量：** 7-10 天

**风险：**
- 参数类型多样化（需要灵活的 UI 组件）


---

### 模块 3：OpenClaw 扩展开发

#### 3.1 任务调度系统

**需求：**
- 批量任务队列
- 并发控制（限制同时生成数）
- 优先级调度
- 失败重试机制

**技术方案：**
- 内存队列 + 简单调度器（MVP）
- Redis + Bull 队列（生产级）
- 数据库持久化任务状态

**可行性：✅ 高**
- 成熟的技术方案
- OpenClaw 基于 Node.js，生态完善

**工作量：**
- MVP 版本：5-7 天
- 生产级：10-15 天

**风险：**
- 并发控制复杂度
- 任务状态一致性

---

#### 3.2 双向同步机制

**需求：**
- OpenClaw → 画布：任务状态推送
- 画布 → OpenClaw：参数变更接收
- 飞书 → OpenClaw：任务创建
- OpenClaw → 飞书：进度更新

**技术方案：**
- 事件驱动架构
- 消息队列（可选）
- WebSocket 双向通信

**可行性：✅ 高**
- 标准架构模式

**工作量：** 7-10 天

**风险：**
- 消息丢失（需要确认机制）
- 状态不一致（需要同步策略）


#### 3.3 素材管理模块

**需求：**
- 本地文件管理
- 云文档上传
- 版本管理
- 素材索引

**技术方案：**
- 文件系统操作（Node.js fs）
- 目录结构规范
- 元数据数据库（SQLite）

**可行性：✅ 高**

**工作量：** 5-7 天

**风险：**
- 磁盘空间管理
- 并发文件操作

---

- 时间轴计算算法

**可行性：✅ 高**

**工作量：** 3-5 天

**风险：**
- 时间轴对齐精度

---

#### 4.3 配乐混音

**需求：**
- 添加背景音乐
- 音量平衡

**技术方案：**
- FFmpeg 音频处理
- 音乐库管理

**可行性：✅ 高**

**工作量：** 3-5 天

**风险：**
- 音乐版权管理

## 二、关键技术风险

### 风险 1：ZeoCanvas API 能力不足 ⚠️ 高风险

**影响：** 如果 ZeoCanvas 没有 API，需要大量二次开发

**应对策略：**
1. 立即调研 ZeoCanvas studio 分支
2. 如无 API，评估添加 API 层的工作量
3. 准备 Puppeteer 备选方案

**优先级：🔴 最高**

---

### 风险 2：飞书 API 频率限制 ⚠️ 中风险

**影响：** 批量操作可能触发限流

**应对策略：**
1. 实现请求队列
2. 使用批量接口
3. 缓存机制

**优先级：🟡 中**

---

### 风险 3：实时同步复杂度 ⚠️ 中风险

**影响：** 多端同步状态不一致

**应对策略：**
1. 设计清晰的状态机
2. 实现冲突解决机制
3. 添加同步日志

**优先级：🟡 中**


---

**关键前提：**

## 五、实施建议

### 建议 1：优先验证 ZeoCanvas

**原因：** 这是最大的不确定性

**行动：**
1. 克隆 ZeoCanvas studio 分支
2. 阅读源码，查找 API 相关代码
3. 尝试启动并测试
4. 评估 API 扩展难度

**时间：** 2-3 天

---

### 建议 2：采用迭代开发

**原因：** 降低风险，快速验证

**策略：**
- 第一轮：单任务流程（2周）
- 第二轮：批量任务（4周）
- 第三轮：画布增强（4周）
- 第四轮：自动剪辑（4周）

---

### 建议 3：技术选型保守

**原因：** 确保稳定性

**选择：**
- 数据库：SQLite（简单）→ PostgreSQL（扩展）
- 队列：内存队列（MVP）→ Redis（生产）
- 部署：单机（初期）→ 容器化（扩展）
